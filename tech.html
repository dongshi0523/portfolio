“替你选”小程序技术可行性说明书

一、 技术架构概述

根据项目流程图，系统主要由前端展示层（微信小程序）、核心业务逻辑层（抽奖算法与筛选）、数据存储层（用户偏好与避雷库）以及第三方集成接口组成。

二、 核心模块可行性分析

1. 抽奖与动画交互模块

技术描述： 实现从首页（SCREEN_33）跳转至筛选页，并展示“大转盘”或“老虎机”形式的抽奖动画（SCREEN_39）。

可行性： 极高。

交互实现： 微信小程序原生支持 Canvas 或 CSS3 动画，能够流畅实现大转盘或老虎机视觉效果。

传感器调用： 流程图提到的“摇一摇”功能（开启摇摇传感器）可通过微信小程序的 wx.onAccelerometerChange 接口监听加速度计数据，触发随机抽取逻辑。

2. “避雷”驱动的动态抽奖算法

技术描述： 系统需整合用户即时筛选条件（分类、预算、距离）与个人中心设置的“避雷库”（如不吃香菜/不吃辣），并反向作用于抽奖结果。

可行性： 高。

逻辑处理： 这属于标准的后端逻辑过滤，可通过数据库查询时的 NOT IN 或 EXCLUDE 操作实现。

实时性： 筛选参数（如 2km 以内、25 元以下）可通过前端实时传递至后端进行候选池计算。

3. 第三方平台深度集成（核心难点）

技术描述： 在最终结果页（SCREEN_4）点击“立即下单”，需直接跳转至美团、饿了么、淘宝或京东等平台的对应商家或菜品详情页。

可行性： 中等（需特定技术方案）。

跳转机制： 微信小程序支持使用 wx.navigateToMiniProgram 跳转至其他小程序。

数据挑战： 获取这些平台的实时菜单和精确的菜品 URL 链接通常需要接入各平台的开放 API（如美团开放平台、联盟 API）或者通过第三方中间件。若无官方接口支持，可能仅能跳转至对应平台的小程序首页或搜索页，无法直接定位到具体菜品。

4. 社区与内容管理系统

技术描述： 提供美食种草与避雷内容的发布流程（SCREEN_10），包括图片上传、评价与链接提交。

可行性： 高。

文件存储： 可使用腾讯云或阿里云的对象存储（OSS/COS）处理用户上传的高清图。

待开发预留： 流程图中标注的“发布编辑器”属于成熟的 CMS 开发范畴。

三、 技术风险评估与对策

四、 结论

从技术角度看，“替你选”小程序的核心功能（随机抽奖、偏好过滤、社区交互）具有很高的开发可行性。项目的最大挑战在于与第三方外卖平台的数据对接和深度跳转路径的打通。

下一步行动建议：

确认目标平台（美团/饿了么等）的开发者入驻权限，获取 API 文档。

建立初步的“美食/标签”数据库结构，以支撑“避雷库”的反向作用逻辑。

开发 MVP（最小可行性产品）版本，先行测试“摇一摇”与本地固定列表的抽奖流程。
